## 一、角色定义

### 1.1 架构师（Architect）

| 维度 | 说明 |
|------|------|
| **身份** | 主 Agent / 人类工程师 |
| **职责** | 需求分析、技术架构设计、任务拆分、接口定义、最终 Code Review |
| **权限** | 读写所有文件、批准合并、否决不达标代码 |
| **产出** | 架构文档、任务清单（TODOS.md）、接口契约、Review 意见 |

### 1.2 开发工程师（Dev Agent × N）

| 维度 | 说明 |
|------|------|
| **身份** | AI 编码子 Agent（可并行启动多个） |
| **职责** | 按任务单实现代码，写单元测试，提交变更 |
| **权限** | 仅读写自己负责的文件/目录 |
| **产出** | 功能代码 + 单元测试 + 简要实现说明 |

### 1.3 QA 工程师（QA Agent）

| 维度 | 说明 |
|------|------|
| **身份** | AI 测试子 Agent |
| **职责** | 编写集成测试、E2E 测试、自动化回归验证 |
| **权限** | 读写测试文件、只读业务代码 |
| **产出** | 测试用例 + 测试报告 + Bug 清单 |

### 1.4 集成工程师（Integrator）

| 维度 | 说明 |
|------|------|
| **身份** | 主 Agent 或专职 Agent |
| **职责** | 合并各分支、运行完整测试流水线、执行部署 |
| **权限** | 合并到主分支、触发部署 |
| **产出** | 合并后的主分支、部署结果、变更日志 |

## 二、完整开发流程

### 2.1 流程总览

```
需求分析 → 架构设计 → 任务拆分 → 接口约定 → 并行开发 → 自动化测试 → 集成验收 → 上线部署
   [架构师]     [架构师]     [架构师]     [架构师]    [Dev×N]      [QA]        [集成]       [集成]
```

### 2.2 各阶段详述

#### 阶段 1：需求分析

| 维度 | 说明 |
|------|------|
| **输入** | 用户需求、现有代码、SPEC.md |
| **动作** | 梳理功能清单、识别现有问题、确认优先级 |
| **输出** | `REFACTOR_PLAN.md`（重构方案） |
| **验收标准** | 覆盖所有功能模块，问题诊断清晰，目标明确 |

#### 阶段 2：架构设计

| 维度 | 说明 |
|------|------|
| **输入** | 需求分析结果、现有代码结构 |
| **动作** | 设计目标目录结构、模块划分、依赖关系图 |
| **输出** | 架构文档（含目录树）、模块依赖图 |
| **验收标准** | 每个模块职责单一、依赖方向清晰、无循环依赖 |

#### 阶段 3：任务拆分

| 维度 | 说明 |
|------|------|
| **输入** | 架构设计文档 |
| **动作** | 拆分为独立任务，每个任务对应一个文件/模块 |
| **输出** | `TODOS.md`（任务清单），每项标注：任务ID、描述、负责文件、预估工时、依赖 |
| **验收标准** | 任务间无文件冲突、依赖关系明确、每个任务 ≤ 2h 工作量 |

#### 阶段 4：接口约定（关键！）

| 维度 | 说明 |
|------|------|
| **输入** | 任务清单、架构设计 |
| **动作** | **先定义所有接口签名，再开始编码** |
| **输出** | 接口契约文件（函数签名、请求/响应格式、错误码） |
| **验收标准** | 每个任务依赖的接口都已明确定义，Dev 可直接参照实现 |

> ⚠️ **接口先行原则**：接口定义是所有并行开发的前提。没有接口，不许开工。

#### 阶段 5：并行开发

| 维度 | 说明 |
|------|------|
| **输入** | 接口契约 + 任务清单 |
| **动作** | 每个 Dev Agent 在独立分支上实现自己的任务 |
| **输出** | 功能代码 + 单元测试 + 提交记录 |
| **验收标准** | 代码通过 lint、单元测试通过、不修改非自己负责的文件 |

**并行开发铁律：**
- 每个 Agent 只碰自己任务清单里的文件
- 公共文件（如 `extensions.py`）由架构师统一编写或指定唯一负责人
- 禁止跨模块调用未定义接口

#### 阶段 6：自动化测试

| 维度 | 说明 |
|------|------|
| **输入** | 各 Agent 提交的功能代码 |
| **动作** | QA Agent 编写集成测试 + E2E 测试，运行完整测试套件 |
| **输出** | 测试报告（覆盖率、通过率、失败用例） |
| **验收标准** | 单元测试覆盖率 ≥ 80%，集成测试 100% 通过 |

#### 阶段 7：集成验收

| 维度 | 说明 |
|------|------|
| **输入** | 所有功能分支 + 测试报告 |
| **动作** | 架构师逐分支 Review → 合并 → 完整回归测试 |
| **输出** | 合并后的主分支、Review 记录 |
| **验收标准** | 无 lint 错误、所有测试通过、功能与需求对齐 |

#### 阶段 8：上线部署

| 维度 | 说明 |
|------|------|
| **输入** | 验收通过的主分支 |
| **动作** | 构建 → 部署 → 冒烟测试 |
| **输出** | 线上可用版本、变更日志 |
| **验收标准** | 线上核心路径可正常访问，无 500 错误 |

## 三、任务拆分规范

### 3.1 文件级隔离原则

```
每个任务 = 一个独立文件（或小文件组）
每个 Agent = 只碰自己的文件
公共文件 = 唯一负责人
```

**✅ 正确拆分：**
```
Dev-A → blueprints/auth/views.py
Dev-B → blueprints/txl/views.py  
Dev-C → blueprints/lyb/views.py
Dev-D → blueprints/pages/views.py
架构师 → extensions.py, config.py, decorators.py, errors.py
```

**❌ 错误拆分：**
```
Dev-A → app.py 前 2000 行
Dev-B → app.py 后 2000 行
```
（同一文件 = 合并冲突 = 浪费时间）

### 3.2 接口先行原则

并行开发前，架构师必须先产出**接口契约**：

```python
# 示例：blueprints/txl/views.py 的接口契约
# blueprint 名称: txl_bp
# URL 前缀: /api/txl

# GET /api/txl/list → 获取通讯录列表
# 参数: page(int, default=1), per_page(int, default=20), keyword(str, optional)
# 返回: {"code": 0, "data": {"items": [...], "total": int, "page": int}}

# GET /api/txl/<user_id> → 获取单条记录
# 返回: {"code": 0, "data": {...}}

# PUT /api/txl/update → 更新个人信息
# 需要登录验证
# Body: {"name": str, "phone": str, ...}
# 返回: {"code": 0} 或 {"code": 401, "message": "..."}
```

Dev 看到这份契约后：
1. 知道要暴露哪些路由
2. 知道入参/出参格式
3. 知道错误码含义
4. 可以直接实现，不用猜

### 3.3 依赖关系管理

```
无依赖任务 → 可立即并行
有依赖任务 → 被依赖方先完成，或先用 Mock 数据开发
```

**依赖处理策略：**

| 场景 | 策略 |
|------|------|
| A 依赖 B 的函数 | B 先定义函数签名（空实现返回默认值），A 参照调用 |
| A 依赖 B 的数据库字段 | 架构师统一定义 schema，A、B 各自使用 |
| A 依赖公共配置 | 架构师先写好 `config.py` |

### 3.4 任务粒度建议

- **最大**：单个任务不超过 2 小时工作量
- **最小**：单个任务不少于 15 分钟（太小了管理成本不划算）
- **理想**：30-60 分钟/任务

## 四、质量控制

### 4.1 自动化测试方案

#### 单元测试（Dev Agent 负责）

```
每个蓝图模块自带测试文件
位置: tests/unit/test_<模块名>.py
要求: 每个公共函数至少 1 个正常用例 + 1 个异常用例
```

#### 集成测试（QA Agent 负责）

```
跨模块接口调用测试
位置: tests/integration/test_<场景>.py
覆盖: 用户注册→登录→修改资料→留言→查看
```

#### E2E 测试（QA Agent 负责）

```
浏览器端到端测试（可用 Playwright）
位置: tests/e2e/test_<页面>.py
覆盖: 核心用户路径（首页访问、留言提交、通讯录查看）
```

### 4.2 Code Review 流程

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  Dev 自评   │ ──→ │ 架构师 Review │ ──→ │  修改/通过   │
│  checklist  │     │  逐项检查     │     │  打回/合并   │
└─────────────┘     └──────────────┘     └──────────────┘
```

**Dev 自评 Checklist：**
- [ ] 代码通过 lint（flake8 / black）
- [ ] 单元测试全部通过
- [ ] 未修改非自己负责的文件
- [ ] 函数有 docstring
- [ ] 异常处理完整（不吞异常）
- [ ] 无硬编码密钥/密码

**架构师 Review 重点：**
- [ ] 接口实现与契约一致
- [ ] 无安全漏洞（SQL 注入、XSS）
- [ ] 代码风格与项目一致
- [ ] 无重复代码（DRY）
- [ ] 错误处理统一

### 4.3 回滚方案

| 场景 | 回滚方式 |
|------|----------|
| 单个分支有问题 | 拒绝合并，打回修改 |
| 合并后发现 Bug | `git revert <commit>` 精准回滚 |
| 整批部署后故障 | 保留上一版本镜像/目录，一键切换 |
| 数据库迁移失败 | 迁移脚本必须支持 rollback |

**铁律：每次合并前，确保主分支可安全回退到上一个稳定状态。**

## 五、工具链

### 5.1 任务分发管理

```
架构师产出 TODOS.md → 按任务 ID 分发给各 Dev Agent
每个 Agent 完成后标记 [x]
阻塞/依赖任务标注 [blocked] 和依赖方
```

**TODOS.md 模板：**

```markdown
## 任务清单

### P0 - 基础设施（架构师负责）
- [ ] T001: 创建 extensions.py [依赖: 无]
- [ ] T002: 创建 config.py [依赖: 无]
- [ ] T003: 创建 decorators.py [依赖: T001]
- [ ] T004: 定义接口契约文件 [依赖: T001-T003]

### P1 - 蓝图实现（Dev 并行）
- [ ] T005: blueprints/pages/ [依赖: T004] [Agent: Dev-A]
- [ ] T006: blueprints/auth/ [依赖: T004] [Agent: Dev-B]
- [ ] T007: blueprints/txl/ [依赖: T004] [Agent: Dev-C]
...
```

### 5.2 代码合并

```
每个任务独立分支: feature/<task-id>
架构师在主分支上合并
禁止 Dev 直接推主分支
```

**分支策略：**
```
main
 ├── feature/T005-pages     ← Dev-A
 ├── feature/T006-auth      ← Dev-B
 ├── feature/T007-txl       ← Dev-C
 └── feature/T008-lyb       ← Dev-D
```

### 5.3 自动化测试流水线

```
提交代码 → 触发测试 → 报告结果
              ↓
    ┌─────────┼──────────┐
    ↓         ↓          ↓
  lint    unit tests   integration
```

**建议工具：**
- `pytest` — Python 测试框架
- `pytest-cov` — 覆盖率报告
- `flake8` — 代码风格检查
- `playwright` — E2E 测试（可选）

## 六、实战案例：HitX 同学录重构

### 6.1 项目背景

**现状问题：**
- `app.py` 4000+ 行，所有路由塞在一起
- `wx_api.py` 1400+ 行，混入非小程序功能
- 无统一错误处理、无统一中间件
- 重复代码多处存在
- 合并冲突地狱

**重构目标：**
- 拆分为 Flask Blueprint 架构
- 按功能模块独立文件
- URL 路由 100% 兼容
- 渐进式迁移，每次迁移后功能可用

### 6.2 具体拆分方案

#### 阶段 1：基础设施（架构师，预计 1h）

| 任务 ID | 文件 | 说明 |
|---------|------|------|
| T001 | `extensions.py` | 扩展初始化（scheduler, db 连接） |
| T002 | `config.py` | 配置集中管理 |
| T003 | `decorators.py` | 登录检查、管理员检查装饰器 |
| T004 | `errors.py` | 统一错误处理 + 错误码定义 |
| T005 | `utils.py` | 通用工具函数（压缩、IP 解析、缓存） |
| T006 | `app.py` 重写 | 应用工厂，120 行以内 |
| T007 | `blueprints/__init__.py` | 蓝图注册入口 |

#### 阶段 2：接口约定（架构师，预计 0.5h）

架构师产出接口契约文件，定义每个蓝图的：
- Blueprint 名称和 URL 前缀
- 所有路由的函数签名
- 请求参数格式
- 响应格式和错误码

#### 阶段 3：并行开发（4 个 Dev Agent 并行，预计 1.5-2h）

| Agent | 任务 ID | 负责模块 | 文件 | 路由数 |
|-------|---------|----------|------|--------|
| Dev-A | T008 | 页面渲染 | `blueprints/pages/views.py` | 9 |
| Dev-B | T009 | 认证模块 | `blueprints/auth/views.py` | 7 |
| Dev-C | T010 | 通讯录 | `blueprints/txl/views.py` | 3 |
| Dev-D | T011 | 留言评论 | `blueprints/lyb/views.py` | 6 |
| Dev-A | T012 | 媒体管理 | `blueprints/media/views.py` | 12 |
| Dev-B | T013 | 喊话/动态 | `blueprints/shout/views.py` | 9 |
| Dev-C | T014 | 小程序 API | `blueprints/wx_miniapp/views.py` | 38 |
| Dev-D | T015 | AI 功能 | `blueprints/ai/views.py` | 6 |

**依赖关系图：**
```
基础设施 (T001-T007) ──→ 接口契约 (T007) ──→ 所有蓝图并行开发 (T008-T015)
                                                    ↓
                                              QA 测试 + 集成
                                                    ↓
                                               合并上线
```

#### 阶段 4：QA 测试（QA Agent，预计 1h）

| 测试类型 | 覆盖内容 |
|----------|----------|
| 单元测试 | 每个蓝图的关键函数 |
| 集成测试 | 用户注册→登录→编辑资料→留言→查看全流程 |
| 兼容性测试 | 所有旧 URL 仍能正常访问 |
| 回归测试 | 对比重构前后的功能清单，逐项验证 |

#### 阶段 5：集成验收 + 部署（架构师/集成，预计 0.5h）

- 逐分支 Code Review
- 合并到 main
- 完整回归测试
- 部署到服务器
- 线上冒烟测试

## 七、核心原则总结

1. **接口先行**：没有接口定义，不许并行开发
2. **文件隔离**：每个 Agent 只碰自己的文件，禁止交叉修改
3. **任务拆分**：粒度 30-60 分钟，最大不超过 2 小时
4. **测试同步**：代码写完，测试紧跟，不裸奔上线
5. **架构师把关**：所有合并必须经过 Code Review
6. **可回滚**：每次变更都能安全回退
